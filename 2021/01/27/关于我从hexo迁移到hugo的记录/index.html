关于我从hexo迁移到hugo的记录

  2021-01-27 0 0  评论 字数统计: 1724字 阅读时长: 4分

1、初步体验

hexo 固然是非常优秀的静态博客生成框架，但是它是基于 node 开发的框架，在打包博客的效率上，实在是不尽人意。我之前 60 多篇文章，全部打包部署完毕，平均时间在 105 秒左右。而且 node 开发的项目，必然是不可能存在小体积的项目，实在是不堪 node_modules 的负重。我的博客环境体积共 80M+，实际占用100M。

我为什么选择了 hugo？

hugo是基于 Go 语言开发的静态博客框架，天生的语言优势在效率上是自带buff的，不出所料，我的30多篇博客（删除了很多质量低下的博客），在仅仅3 秒钟内就已经打包部署完毕了。而且体积上，我的博客工作环境的大小，还不到3M。

总结一下

hexo

  • 优点
    • 生态环境好，用户基础多
    • 使用体验上，几乎是无可挑剔
    • 主题丰富，选择众多
  • 缺点
    • 体积过大，不可避免
    • 打包效率很低，不可避免

hugo

  • 优点
    • 打包速度快
    • 体积小
    • 适合博文多的人群
  • 缺点
    • 生态远不如hexo 好
    • 用户基础比较差
    • 需要一定的动手能力

2、踩坑

我个人目前使用的博客主题，魔改自 hugo-theme-pure。

1、整体字体调整

整个主题的字体都是偏小的，我个人看起来是十分的不舒服，因为我需要挤眼睛才能看的稳。这样对眼睛太不友好了，于是我在 issues 打算查查，没想到也有小伙伴跟我同感。在大佬的指点下，

将 body 中的 font-size，由基础的14px 调整到 15.5px

2、代码块调整

原始的代码块，那个字体看起来很糊，也很小。实在是不符合我个人的使用。

考虑到，代码块使用的是 highlight 处理的，因为我直接将问题定位如何修改 highlight 代码块的字体。

果不其然，我找到了最终要修改的 css 属性。

这里的 css 样式是要放在 put custom.css in $hugo_root_dir/static/css。根据作者的注释，可以知道，在加载时，会自动扫描此包下的所有 css 。

pre > code {
    font-family: "Sans Mono", "Consolas", "Courier", monospace;
    font-size: 15.5px;
}

同样的，我修改成了和 body 等值。这样整体的阅读障碍就彻底消失了。

Consolas 字体是我最喜欢的一款字体，在 Jetbrains Mono 没推出前，现在用 Jetbrains Mono用喜欢了，而且 Jetbrains 默认就是这个，本身也不错，再加上懒得修改字体。

3、首页文章列表标题的调整

现在的效果如下，显得很突出。

这个很容易就可以修改，打开控制台

把 css 的类名复制一下，到 themes/assets/css/style.css 里，找这个选择器，改成 1em，或者自己感觉舒服的大小就行。建议是 em 单位，这样的话，移动端也会适配。

4、文章目录不见了？

这是因为，作者没有写这个，因此，我们需要自己添加。

在目录 themes\pure\layouts\partials\sidebar-toc.html 下

修改如上图所示，在 <nav></nav> 中，添加 {{ .TableOfContents }}。这是 hugo 默认的文章目录设置。

那么此时，如果很不幸，你的文章不包含任何的二级标题，那么此时文章目录仍然是无法显示的。

中午找了一个小时的答案，最终在 hugo 论坛，找到了答案。Why TOC is not working?

In the TOC - only the “Third H2” is listed - the other ones are ignored.

意思是，只有二级标题及其下面的小标题，才能够被正确录入。

3、对hugo的个人理解

在hugo里，hugo new xxx.md 都是一个页面。一个 md 对应一个页面。

总结一下，个人用得到的一些配置信息，基于个人使用的主题

在 $hugo_root/archetypes/default.md，是默认生成的新页面的配置信息，我的个人配置如下

# 文章标题
title: "{{ replace .Name "-" " " | title }}"
# 文章创建日期
date: {{ .Date }}
# 是否为草稿
draft: true
# 作者
author: "Hugo Authors"
# 文章简介
description: "Hugo, the world’s fastest framework for building websites"
# 是否启用 数学公式渲染
math: true
# 是否开启文章目录
toc: true
# 文章访问 url
url: /

4、分享一个自动部署的脚本

hugo 编译打包后，会在 $hugo_dir_root 下生成一个 public 文件夹。我们要做的，就是将 public 里的所有内容，全部打包上传到 gitee / github 上的一个 gitee / github pages 服务的仓库。

hugo 和 hexo 不同，hexo 是一键式操作，同时支持自动部署等。hugo就需要我们自己写一个脚本来实现自动部署的效果，下面是我的目录结构图。建议根据自己的实际情况来编写脚本即可。

目录示意图：

仅供参考

#!/bin/bash

# 博客发布项目仓库 和 博客源码项目必须在同级目录

start_time=`date --date='0 days ago' "+%Y-%m-%d %H:%M:%S"`
echo "hugo 打包文件中...."

# 删除 除 .git目录和commit.sh 之外的所有文件和文件夹
rm -rf `ls | egrep -v '(.git|commit.sh)'`

# 先进入 my_blog 博客源代码目录
cd ..
cd my_blog/

# 开始提交源代码
cur_time=$(date "+%Y-%m-%d %H:%M:%S")
git add .
git commit -m "auto commit on $cur_time"
git push
# 源码保存完毕
echo "源码保存完毕"

# 删除所有内容
rm -rf public/*

# 编译
hugo

# 将 public 下所有文件 全部转移到 qybit下
mv ./public/* ../qybit/

cd ..
cd qybit/


git add .
git commit -m "auto commit on $cur_time"
git push

finish_time=`date --date='0 days ago' "+%Y-%m-%d %H:%M:%S"`
duration=$(($(($(date +%s -d "$finish_time")-$(date +%s -d "$start_time")))))
echo "共计耗时：${duration}秒"
